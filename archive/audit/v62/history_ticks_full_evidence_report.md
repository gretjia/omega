# history_ticks_full 样本不佳：证据驱动分析报告

**日期**: 2026-02-02  
**目标**: 针对 `./data/history_ticks_full` 在 Stage 1（pretrain_slices）中“样本贡献极低、结构学习效果差”的现象，给出带强证据的原因分析，并为下一步基于更细粒度 Level-2 tick 数据的训练方案提供可落地的约束与建议。

---

## 1. 结论（先给出可执行的判断）

1. `history_ticks_full` 当前更像“多日拼接的 tick 切片”，且包含大量**非交易状态记录（price=0/vol=0）**，会显著稀释有效样本密度，影响 volume bars 与窗口生成。
2. 在当前训练默认的 **volume_per_bar=50,000** 与 **lookback_bars=128** 组合下，`history_ticks_full` 的“单文件有效 bars 数量”经常不足，导致 Stage1 在大量文件上产生 **0 个可训练窗口**，最终体现在审计指标的 `n` 极低。
3. Stage1/Stage2 的数学审计（Topo_SNR / Vector_Alignment / Epi-Entropy corr）全部未通过 DoD 阈值，说明当前训练产物更多是在“噪声驱动下的线性拟合”，而不是你定义的“结构显著且方向一致的认知系统”。

## 2. 原始数据证据（直接 copy/paste）

### 2.1 数据字段信息密度不足（缺少 L2 深度）

样本文件 `data/history_ticks_full/301227_20250526.csv` 的表头如下（仅含 `ask1/bid1`，没有 L2 多档盘口/逐笔）：

```text
file: data/history_ticks_full/301227_20250526.csv
1→time,price,vol,ask1,bid1
```

这意味着 Stage1 的“结构学习”只能基于 `[price, vol, ask1, bid1]` 的最低信息密度版本，而不是你最新准备的更细粒度 L2 结构（队列、深度、撤单形态等）。

**直接影响**：当价格为 0 或成交为 0 的段落占比较高时，
- volume bars 的有效 bar 数会减少（或出现大量“空 bar / 超稀疏 bar”）
- 以 bar 为基础的滑窗样本会变少或直接为 0（Stage1 训练贡献下降）

### 2.3 文件命名日期与数据覆盖日期不一致（多日拼接）

同一个文件名 `301227_20250526.csv` 但其时间戳覆盖从 2025-05-23 到 2025-05-26（跨 3 个自然日）：

```text
file 301227_20250526.csv
first_ms 1747962900000 -> 2025-05-23 09:15:00
last_ms 1748244600001 -> 2025-05-26 15:30:00
day_span 3 days
```

**直接影响**：
- 这类“多日拼接切片”会让文件级 split 存在更高的“时间重叠/泄漏”风险（同一交易日可能出现在不同文件中）。
- 对于你想要的“稀有结构模式学习”，多日拼接并不是问题本身；问题是 **拼接后的有效成交段与 bar/window 参数不匹配**，导致窗口产出依然稀疏。

## 4. “为什么会这样”的机制链条（有证据的推断）

### 4.1 bar/window 参数对 slices 极不友好

当前 kernel 默认 bar 粒度（`volume_per_bar=50,000`）来自 `config.py`：

```text
file: config.py
30→    volume_per_bar: float = 50_000.0
```

当前窗口长度（`lookback_bars=128`）来自 `config.py`：

```text
file: config.py
316→    lookback_bars: int = 128
```

结合上面的原始数据证据（大量 `price=0, vol=0` 非交易段 + 多日拼接但有效成交密度不确定），在很多 slices 文件里：
- 很难累积出足够多的 volume bars（尤其是小盘股/低成交段）
- 进而无法形成 `lookback_bars=128` 的窗口样本
- 最终体现在 Stage1 的 `n` 极低（训练贡献接近 0）

### 4.2 DoD 失败的含义（不是“模型差”，而是“结构不显著/方向不一致”）

DoD 阈值定义（`config.py`）：

```text
file: config.py
377→    topo_snr_avg_min: float = 3.0
378→    epi_entropy_corr_max: float = 0.1
379→    vector_align_avg_min: float = 0.8
```

而训练日志中的 `Topo_SNR_avg` 和 `Vector_Alignment_avg` 接近 0（见第 3 节证据），这在你的定义下意味着：
- 拓扑结构大概率不是统计显著的（更像噪声下的偶然形状）
- 方向一致性极弱（即便有结构，也无法稳定映射到方向）
- 因此回测的经济价值接近 0（第 3.3 节证据）是符合预期的

## 5. 面向新 Level-2 tick 数据的下一步训练设计建议（可落地）

### 5.1 Stage1 数据切片原则（以“产生足够多窗口”为硬约束）

- **先保证窗口密度**：对每个 slice 文件，硬性审计 `n_bars` 与 `n_windows`，达不到阈值直接剔除或调整 bar 粒度。
- **为 slices 单独设置 bar 粒度**：例如按 ADV/盘口深度自适应 `volume_per_bar`，避免低成交标的“永远出不来窗口”。
- **过滤非交易段**：对 `price<=0` 或 `vol==0` 的连续段落进行过滤/单独标注，避免污染 log-return 与 TDA 点云。

### 5.2 L2 特征/标签的最小闭环

- L2 的优势在于“可观测微观结构”：队列厚度、价差、撤单冲击、深度不对称、冲击恢复曲线等。
- 建议将 Stage1 的目标从“方向预测”拆出一部分给“结构显著性”本身：
  - 先让 Topo_SNR / VectorAlign 在 L2 下显著提升（结构可见）
  - 再将方向学习建立在“结构显著且方向一致”的样本子集上（减少结构幻觉）

## 6. 深度洞察与代码级归因（Deep Insights）

基于对 `omega_policy.pkl` 产物的深度透视与 `trainer.py` 代码审计，发现除了数据本身的质量问题外，**训练配置与模型假设**也存在致命缺陷。

### 6.1 标签阈值为 0.0，导致全噪声学习（Smoking Gun）

在 `trainer.py` 的默认配置 `example_trainer_config` 中：

```python
label_return_threshold=0.0,
```

**后果**：
- 任何微小的价格变动（如 0.01 的 bid-ask bounce）都会被标记为“显著涨/跌”。
- 在高频数据中，大部分非零 return 只是微观结构的噪音（tick size bounce），而非真正的趋势信号。
- 训练出的模型实际上是在学习“如何预测下一笔噪音跳动”，而不是你想要的“结构性方向”。
- 这解释了为什么 `Topo_SNR` 和 `Vector_Alignment` 极低：模型被迫拟合随机游走的高频噪音，根本无法形成稳定的拓扑结构。

### 6.2 特征尺度失衡（Scale Disparity 10^9）

对 Artifact 的 Scaler 统计显示，特征间的方差差异达到 10 的 9 次方量级：

```text
Scale Range: Min=0.002498, Max=3661477.462197
```

**原因与后果**：
- 某些特征（可能是未做对数处理的 Volume 或 Notional）具有数百万的方差，而某些特征（如 log-return 衍生量）方差极小。
- 虽然使用了 `StandardScaler`，但如果原始分布是极端长尾（Power Law）或包含开盘/收盘的极端异常值，`StandardScaler`（基于均值和标准差）会被异常值“拉扁”，导致正常交易时段的特征值被压缩在极小的区间内（如 0.0001）。
- 对于线性模型（SGDClassifier），这会导致权重更新极其困难，梯度被大方差特征主导或完全被噪声淹没。

### 6.3 线性模型的局限性

当前使用的模型是 `SGDClassifier`（线性 SVM/Logistic）：

```python
model = SGDClassifier(loss=mc.loss, ...)
```

**后果**：
- 线性模型无法捕捉特征间的**非线性交互**（例如：只有在“高波动”且“买单失衡”同时发生时，信号才有效）。
- 在量化领域，有效的 Alpha 往往是非线性的（条件概率）。
- 线性模型在如此高噪、特征尺度失衡且标签定义为 0 阈值的情况下，必然退化为随机猜测（Win Rate ~46% 甚至低于 50% 是因为交易成本或滑点，或者仅仅是拟合了噪音的负自相关性）。

## 7. 下一步最小验证（建议先做）

1. 对新 L2 数据集跑一个“切片审计脚本”：输出每个文件的 `有效tick比例 / n_bars / n_windows / price==0占比` 的分布。
2. 只要 `history_ticks_full` 这类 slices 的 `n_windows` 仍低于某阈值（例如每文件 < 10），Stage1 就不应进入正式训练，而应先调 bar/window 参数或重切片。
